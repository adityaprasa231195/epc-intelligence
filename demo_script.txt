orders are in another. RFI answers are buried in email chains.

We built the EPC Intelligence Platform to fix this. It is a
unified AI layer over your entire project — specs, schedules,
supply chain, commissioning records, and RFIs — all connected,
all queryable in plain English. Let me show you how it works."

SECTION 3 — ARCHITECTURE
[ Target: 30 seconds ]
================================================================

(Optional: show architecture diagram if printed or on slide)

Say:

"Let me quickly walk through what is under the hood.

We have five specialist AI agents — each owning one domain:
compliance, schedule, supply chain, commissioning, and knowledge.

Above them sits the Orchestrator — it routes any plain-English
question to the right agent, first by keyword matching, and if
that fails, it asks the LLM to classify the intent.

All agents run on Groq — one of the fastest LLM inference
platforms available. Text generation uses Llama 3.3 70B.
Drawing analysis uses Llama 4 Scout with vision capability.

The knowledge layer is a RAG engine — it chunks the TIA-942
standard and all project RFIs, retrieves the most relevant
sections, and feeds them into the LLM as context. Every answer
has a citation.

The entire application runs on four Python packages. No heavy
ML frameworks. No cloud databases. Fast, lean, and deployable
in under 30 seconds."

BACKUP ANSWERS FOR Q&A
================================================================

Q: "Is this using real data or synthetic data?"
A: "The demo uses synthetic project data we built to represent
   a realistic Mumbai data centre EPC project — procurement
   items, schedule, supplier locations, and RFIs. In production,
   you replace the JSON files with live feeds from your project
   management system. The agents do not change."

Q: "How accurate is the RAG system without a vector database?"
A: "For exact specification lookups — which is the primary use
   case here — keyword retrieval on a controlled corpus of
   standards is highly reliable. We also inject the full
   document context into Groq's 131,000 token context window,
   so the LLM sees everything, not just the retrieved chunks.
   In production we would swap in a vector database for larger
   corpora — it is a single file change."

Q: "How long did this take to build?"
A: "The core platform — all five agents, the orchestrator, the
   RAG engine, and the UI — was built during the hackathon
   period. The architecture is modular by design, so each agent
   can be extended independently."

Q: "Can this integrate with existing project management tools?"
A: "Yes. Each agent loads data through a single _load method.
   Replacing the JSON file with a Procore API call or a P6
   schedule export is a straightforward swap. The AI layer
   above it does not change."

Q: "What happens if Groq is down or rate-limited?"
A: "Every single LLM call has a deterministic fallback. If Groq
   returns an error, the system falls back to pre-built data —
   curated mitigation lists for schedule, approved vendor lists
   for supply chain, expert commissioning checklists for QA.
   The app never crashes and never shows a blank screen."
